I/O

5 October 2019

在 Linux 系统中一切皆可以看作是文件，文件可分为：普通文件，目录文件，链接文件和设备文件等。文件描述符（file descriptor）是内核为了高效管理已被打开的文件所创建的索引，索引是一个非负整数（>=0），用于指代被打开的文件，所有执行 I/O操作的系统调用都通过文件描述符。可执行程序刚启动时，0是标准输入，1是标准输出，2是标准错误。如果此时去打开一个新文件，那么它的文件描述符就为3。POSIX 标准要求每次打开文件时（含 socket）必须使用当前进程中最小可用的文件描述符号码，因此，在网络通信过程中稍不注意就有可能造成串话

文件描述符的限制

在编写文件操作或者网络通信的软件时，有时候会遇到“Too many open files”的问题。这主要是因为文件描述符是系统的一个重要资源，虽然说系统内存有多少就可以打开多少的文件描述符，但是在实际实现中内核是会做相应的处理的，一般最大的文件打开数会是系统内存的10%（称为系统级限制），通过 sysctl -a | grep fs.file-max 可以查看（结果的单位是 KB）。与此同时，内核为了不让某一个进程消耗掉所有的文件资源，会对单个进程可以打开的文件资源做一个限制（称为用户级限制），默认为1024，使用 ulimit -n 命令可以查看

文件描述符和文件的关系

每一个文件描述符会与一个打开的文件相对应，同时，不同的文件描述符也可能会指向同一个文件。相同的文件也可以被不同进程打开，也可以在同一个进程中被多次打开。系统为每一个进程维护了一个文件描述符表（表的大小与上面说的用户级限制相关联），该表的值从0开始，所以在不同进程中你会看到相同的文件描述符，而且可能指向的是同一个文件

open 和 fopen

这里不是要讲 open 和 fopen 这两个函数，而且要讲系统调用（提供了 open）和 C 标准库（提供了 fopen）的区别。当你使用 C 标准库的相关 I/O 调用时，会返回一个 FILE 结构体指针，我们讲过进程启动后默认会有3个文件描述符（0，1，2），同样默认也有3个 FILE 结构体指针对应到3个文件描述符，它们是 stdin，stdout 和 stderr

fopen 是基于 open 实现的，当我们使用 fopen 打开文件时会返回 FILE 结构体指针，它包含了文件描述符，读写内容的游标和 buffer cache （一般大小为 8kb）等。当你使用 fopen 操作文件时，它不会直接将数据写入到对应文件中，而是暂缓在 buffer cache 中，只有下面3种情况才会将数据回写到对应的文件中

  1. buffer cache 满了
  2. 调用 fclose，释放了文件描述符和 buffer cache
  3. 调用 fflush，手动释放了 buffer cache

通过下面这个例子了解 open 和 fopen

#include <unistd.h>
#include <sys/types.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <stdio.h>
#include <pthread.h>
#include <stdlib.h>
#include <string.h>
#include <errno.h>

int main(int argc, char const *argv[])
{
    // 关闭了标准输出流 1
    close(STDOUT_FILENO); 
    // 通过系统调用打开一个文件，由于之前关闭了标准输出流，所以现在当前进程中最小可用的文件描述符就是1
    int fd = open("1.log", O_CREAT|O_TRUNC|O_WRONLY, 0644);
    // printf 底层是实现是 fwrite，往 stdout 写数据
    printf("hello world\n"); 
    // 手动释放 FILE 结构体指针 stdout 的 buffer cache
    // 你可以试试有没有这行代码会有什么区别，看 1.log
    fflush(stdout); 
    // 只是释放文件描述符
    close(fd);

    return 0;
}

这一行的代码 O_CREAT|O_TRUNC|O_WRONLY 需要说明下，open 的第二个参数是一个 int 类型（32位）

前三位 O_RDONLY，O_WRONLY，O_RDWR 是必选项且只能选一个，其他为可选项，通过 | （或操作）置为1

常用系统 I/O 调用

打开文件 open

int open(const char * pathname, int flags, mode_t mode);

一些常用的 flag 如下

关闭文件描述符 close

int close(int fd)

如果失败的话，返回-1，失败原因会写在 errno 中

读内容 read

ssize_t read(int fd, void* buf, unsigned int size)
  • fd 文件描述符
  • buf 缓冲区对象，也就是读取时候的数据就是存在这个对象中
  • size 缓冲区大小，也就是说要读取多大的数据

函数返回值是实际读取的数据大小。如果失败返回 -1，失败原因会写在 errno 中

写内容 write

ssize_t write(int fd, const void* buf, unsigned int size)
  • fd 文件描述符
  • buf 缓冲区对象，也就是保存数据的对象
  • size 要写多少字节大小的数据

函数返回值是实际写入的数据大小，如果失败返回 -1，失败原因会写在 errno 中

修改游标(移动文件读写位置) lseek

__msvcrt_long lseek(int fd, long off, int where)
  • fd 文件描述符
  • off 设置游标的偏移量
  • where 从哪里开始偏移，有三个宏定义可以用
    • SEEK_SET 开始位置
    • SEEK_CUR 当前位置
    • SEEK_END 结尾位置

函数返回值为返回当前位置到开始位置到长度，如果失败返回 -1，失败原因会写在 errno 中

常用 C 标准库 I/O 调用

打开文件 fopen

FILE * fopen(const char * path, const char * mode);
  • path 文件路径
  • mode 文件打开方式

常用 mode

函数返回值为 FILE 指针，如果失败返回 NULL，失败原因会写在 errno 中

关闭文件 fclose

int fclose(FILE *);

函数返回值为0则表示成功，如果失败返回 EOF，失败原因会写在 errno 中

读内容 fread

size_t fread(void *buffer, size_t size, size_t count, FILE *);
  • buffer 接收数据的地址
  • size 要读取的字节数，单位字节
  • count 要读取多少个 size

函数返回实际读取的单元个数。如果小于 count，则可能文件结束或读取出错。可以用 ferror 检测是否读取出错，用 feof 函数检测是否到达文件结尾。如果 size 或 count 为0，则返回0

你可能会好奇为啥要分 size 和 count 呢？怎么不弄成一个参数传入 size * count 就可以了。其实这种实现方式主要还是从调用者方便考虑的，比如说，你要读写一个 struct 数组，通过返回值，你就能实际知道读写成功多少个 struct，虽然感觉也没方便多少。如果你考虑的是数据是否完全读写成功的话，那么其实也就不用纠结这两个参数的联系了

写内容 fwrite

size_t fwrite(void * buffer, size_t size, size_t count, FILE *);
  • buffer 数据的地址
  • size 要写入的字节数，单位字节
  • count 要写入多少个 size

函数返回实际读取的单元个数。如果小于 count，则说明发生了错误，文件流错误标志位将被设置，随后可以通过 ferror 函数判断

修改游标(移动文件读写位置) fseek

int fseek(FILE *, long off, int where)
  • off 设置游标的偏移量
  • where 从哪里开始偏移，有三个宏定义可以用
    • SEEK_SET 开始位置
    • SEEK_CUR 当前位置
    • SEEK_END 结尾位置

清空文件缓冲区 fflush

int fflush(FILE *)

函数成功返回0，失败返回 EOF，错误代码存于 errno 中
